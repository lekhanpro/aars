Adaptive retrieval that plans, retries, and cites before it answers.

AARS is a query-aware RAG backend that chooses retrieval strategy per request, re-checks evidence through reflection, and fuses keyword, vector, and graph results into grounded answers. The benchmark in this repo is reproducible offline and the runtime path is actually wired end to end.

Capabilities

Built to avoid fixed-pipeline RAG failure modes.

The repo now matches the architecture it describes. These are implemented runtime behaviors, not presentation-layer claims.

01

Planner-backed strategy selection

Queries can route into keyword, vector, graph, hybrid, or none, with request-level toggles and deterministic fallback planning when live LLM planning is disabled.

02

Shared ingestion and query runtime

Startup now builds a shared orchestrator plus ingestion pipeline so collections update Chroma-compatible storage, BM25 state, and graph state together.

03

Reflection before final answering

AARS can evaluate evidence sufficiency, retry retrieval with a revised strategy, then continue into fusion and grounded answer generation.

04

Reproducible offline benchmark

The checked-in benchmark is small by design: 12 local documents, 9 questions, 7 systems, no dataset download, no hosted services required.

05

Graceful dependency fallback

If sentence-transformers or spaCy are unavailable, the repo still runs with hashing embeddings and lightweight entity extraction instead of crashing.

06

Debuggable outputs

Query responses can include the retrieval plan, reflection steps, returned documents, citations, and a bounded execution trace.

Pipeline

What happens per query

  1. Plan Choose retrieval strategy from the query itself, or use a deterministic fallback.
  2. Retrieve Run vector, keyword, graph, or hybrid retrieval against the selected collection.
  3. Reflect Check whether the current evidence is sufficient before spending another answer call.
  4. Fuse Merge ranked lists with reciprocal rank fusion and optionally rerank with MMR.
  5. Generate Return an answer with citations, confidence, documents, and optional trace output.

    Ship It

    Quick start without guessing what still works.

    The repo supports three practical paths: run the benchmark, start the API, or launch the Streamlit UI against a local backend.

    Run the offline benchmark

    python benchmarks/runner.py --output benchmarks/results_local.json

    Start the API locally

    docker run -p 8001:8000 chromadb/chroma:latest
uvicorn src.main:app --host 0.0.0.0 --port 8000 --reload

    Launch the UI

    set AARS_API_URL=http://localhost:8000
streamlit run ui/app.py

    API surface

    POST

    /api/v1/ingest

    Load a text or PDF file into a target collection and update all retrievers.

    POST

    /api/v1/query

    Run planning, retrieval, reflection, fusion, reranking, and answer generation.

    GET

    /api/v1/health

    Report API version and current Chroma connectivity state.

    GET

    /api/v1/collections

    List available collections for the currently configured Chroma backend.

    DELETE

    /api/v1/collections/{name}

    Delete a collection when the backend is available.

    Current limits

    What this repo still does not claim.

    No external benchmark claims

    The checked-in result is the local fixture benchmark only. That is intentional, and the page now says so plainly.

    No fake hosted live demo

    GitHub Pages is static. This site now ships a working sample explorer instead of pretending a public API endpoint exists.

    Live answering still needs an API key

    The offline benchmark runs without a hosted model, but the live API answer path still depends on an Anthropic-compatible key.
